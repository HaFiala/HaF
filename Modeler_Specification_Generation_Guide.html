Modeler Guide: Specification Generation

The generator (archimate_spec_v3.py) reads an ArchiMate XML model and produces a Confluence-formatted HTML spec for a named ApplicationFunction element. This guide describes what to model and how to document elements and relations so the output is correct.

1. The Target Element

The generator is run against a single element by name. Its documentation field becomes the body of the spec. Everything else (relations, child elements) is derived from what that element connects to.

Any element type can be targeted. When multiple elements share the same name, the generator prefers ApplicationFunction — otherwise it takes the first match found in the model.

2. Element Documentation — Structure

The documentation field is rendered line by line. Several lines have special meaning:

Line content What it renders
Description: some text Bold Description: label followed by the text
technically (alone on a line) Bold Technically: label
<loopLabel> (alone, any word) Italic loop/iteration label; body below is indented +40 px
</loopLabel> (alone) Closes the loop; indentation returns
˝relationName˝ Replaced with an inline rendered relation line (see section 4)

All other lines are plain text, rendered with inline token colorization (see section 3).

3. Inline Text Tokens

Inside any documentation text (element or relation), these patterns are colorized automatically:

Pattern Example Rendered as
{variableName} {customerId} Green italic monospace
'UPPERCASE_VALUE' 'ACTIVE' Magenta — must start with uppercase letter; only A–Z, 0–9, _, (, ), : allowed
[type name] [table CUSTOMER] Blue monospace — type must be one of: register, table, view, file, select, definition, folder

Note: lowercase constants like 'active' are not colorized. The constant pattern requires the value to start with an uppercase letter.

4. Abbreviation Mechanism — Embedding Relations in Description

A relation can be rendered inline within the element description by placing ˝relationName˝ (double acute accent, U+02DD) at the desired point in the text. The generator replaces the token with the formatted relation line.

Requirements: - The relation must have its name attribute set in the model — the token value must match the relation name exactly. - The relation's source must be the element being documented.

Gap for modelers: Archi does not display relation names prominently. Setting the name field on a relation is easy to forget. If the abbreviation token does not resolve, the raw ˝text˝ appears verbatim in the output.

Relations rendered via abbreviation are excluded from the Relations section at the bottom of the spec. If all outgoing relations are embedded this way, the Relations section shows "All relations already described above."

5. Access Relations

An AccessRelationship from the function to a Data Object represents a data interaction. How it renders depends on the profile and the relation's documentation.

5.1 Determining the label

Condition Label
Profile set Profile name (used as-is)
No profile, no accessType write
No profile, accessType = 1 read
No profile, accessType = 2 write
No profile, accessType = 3 read/write

The direction word appended to the label is from for read and select; into for everything else.

Generic rendering (no special profile): {label} {from|into} {Target Name} followed by relation documentation in gray.

5.2 Profile: SELECT

Set the relation's profile to SELECT (case-insensitive). The relation documentation drives the output.

Tag: <sellist>...</sellist> — exact lowercase.

Output structure:

select
    {each line of sellist, indented}
from  {Target Name}
    {rest of relation doc, indented, gray}

Documentation example:

<sellist>
{id}, {name}, {status}
</sellist>
WHERE {tenantId} = :tenantId
AND {status} = 'ACTIVE'

Generated output:

select
    {id}, {name}, {status}
from  CUSTOMER_PROFILE
    WHERE {tenantId} = :tenantId
    AND {status} = 'ACTIVE'

Text outside the <sellist> tag (before or after) becomes the "rest of doc" — rendered below the from line, indented and gray.

5.3 Profile: UPDATE

Set the relation's profile to UPDATE (case-insensitive). The relation documentation drives the output.

Tag: <updList>...</updList> — mixed case, exact spelling.

Output structure:

update  {Target Name}
    {each line of updList, indented}
    {rest of relation doc, indented, gray}

Documentation example:

<updList>
{status} = :newStatus
{updatedAt} = SYSDATE
{updatedBy} = {userId}
</updList>
WHERE {id} = :id

Generated output:

update  CUSTOMER_PROFILE
    {status} = :newStatus
    {updatedAt} = SYSDATE
    {updatedBy} = {userId}
    WHERE {id} = :id

5.4 [select] Named Target

For verbatim SQL or complex queries that don't fit the SELECT profile pattern, create a Data Object whose name starts with [select] — for example [select] GetActiveCustomers.

  • The Data Object's own documentation is rendered as a monospace preformatted block (verbatim, no colorization).
  • The relation's documentation is appended line by line after it, with inline colorization.
  • The relation's profile and accessType are ignored — the [select] name prefix takes precedence.

Use this when the full query text needs to be preserved exactly, including multi-line SQL.

5.5 Tag casing — known inconsistency

Tag Required casing
<sellist> all lowercase
<updList> mixed case — capital L

This inconsistency is known and will be corrected. Until then, use the exact casings above.

6. Other Relation Types

These are shown in the Relations section of the spec (or inline via abbreviation). They carry no XML tag conventions — only the relation documentation field is used.

Relation type Outgoing label Incoming label
CompositionRelationship composed of is a part of
TriggeringRelationship triggers is triggered by
ServingRelationship is used by use
AssignmentRelationship* provides is implemented in

* Only when source is ApplicationComponent / SystemSoftware / TechnologyService and target is ApplicationFunction / ApplicationProcess.

Relations with labels composed of, is a part of, triggers, is triggered by, provides, is implemented in, use, is used by are grouped in the Relations section — the label appears once, followed by a bulleted list of names. Access relation labels (select, update, write, read, read/write, custom profiles) are rendered individually with their documentation.

7. Composition to Same-Type Element

A CompositionRelationship from an ApplicationFunction to another ApplicationFunction causes the child's full spec to be embedded inline in the parent spec (recursively). This is triggered via the abbreviation mechanism: place ˝relationName˝ in the parent's description at the point where the child spec should appear.

If the composed target is a different element type, it is rendered as a plain object reference instead.

8. Quick Checklist

  • [ ] If multiple elements share the same name, ensure the intended one is ApplicationFunction (preferred) or is the first occurrence in the model
  • [ ] Access relation has correct profile set (SELECT / UPDATE / or none for write/read)
  • [ ] <sellist> tag is all lowercase; <updList> tag has capital L
  • [ ] [select] Data Object has the full query in its own documentation field
  • [ ] Any relation referenced via ˝abbreviation˝ has its name field set on the relation
  • [ ] Inline constants use uppercase (e.g. 'ACTIVE', not 'active')
  • [ ] Object references use a valid prefix: [table …], [view …], [register …], etc.
